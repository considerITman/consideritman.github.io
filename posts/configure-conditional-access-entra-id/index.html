Complete Guide: Conditional Access Policies in Entra ID

Step-by-step guide to Conditional Access in Microsoft Entra ID. Covers policy anatomy, grant controls, sign-in risk, user risk, and safe rollout for E3 and E5.
May 22, 2026 · 16 min · Alexander Schirmer

Every Microsoft 365 E3 organisation already holds the licence to configure powerful Conditional Access (CA) policies today. Entra ID P1 is included in M365 E3, and it covers named locations, authentication strengths, device compliance grants, and app protection controls. That’s a real Zero Trust foundation, not just an MFA toggle. What E5 adds is automated, continuous risk response: Entra ID Protection feeds real-time sign-in and user risk signals directly into CA policy decisions. This guide walks through the full feature set, from policy anatomy to safe rollout, so you can build the right policies for the licences you hold now and make a clear-eyed decision about upgrading.

Key Takeaways

  • Conditional Access requires Entra ID P1, included in Microsoft 365 E3 and Business Premium.
  • Risk-based policies (sign-in risk, user risk) require Entra ID P2, available in Microsoft 365 E5 or as an add-on.
  • “Require approved client app” was retired in March 2026. Migrate existing policies to “Require app protection policy.”
  • Microsoft recommends at least one week in report-only mode before enforcing any policy (Source: Microsoft Learn, retrieved 2026-05-22).
  • CA App Control requires a separate Microsoft Defender for Cloud Apps licence, not included in E3 or standalone P1/P2.

What Is the Conditional Access Policy Model?#

Conditional Access uses a two-phase evaluation model for every sign-in attempt. Phase 1 collects session details: user identity, device state, location, client application, and the target resource. Phase 2 enforces the policies that match those collected signals. Assignments between categories use AND logic, so a user must satisfy all configured conditions for a policy to apply. Within grant controls, the default behaviour is “require all selected controls.” A single tenant can hold up to 240 policies across all states (enabled, disabled, and report-only). Source: Microsoft Learn, “What is Conditional Access”, retrieved 2026-05-22.

Grant controls are evaluated in a fixed order when multiple grants are required: MFA, device marked as compliant, hybrid join, approved client app, app protection policy, password change, terms of use, then custom controls. Understanding that order matters when you stack controls. If any Block policy applies to a sign-in, it wins regardless of other grant policies in the chain. Keep your scope precise. A broad Block policy will catch more users than intended.

What Do You Need to Set Up Before Configuring Policies?#

Two things must be in place before you touch the policy blade: break-glass accounts and report-only mode discipline. Break-glass accounts are cloud-only Global Administrator accounts excluded from every Conditional Access policy without exception. They exist to recover access if a misconfigured policy locks out all admins. Exclude them by account object, not by group, because group membership can change.

Report-only mode is mandatory as the first state for any new policy. It runs full policy evaluation and logs results without enforcing any action. Outcomes appear under Sign-in logs > Report-only and carry one of four states: Success, Failure, User action required, and Not applied. Microsoft recommends at least one week in report-only before switching to enforcement. Source: Microsoft Learn, “Conditional Access report-only mode”, retrieved 2026-05-22.

Two caveats worth knowing early. The CA Insights workbook (which visualises report-only trends over time) requires Azure Monitor integration. Also, device compliance evaluation in report-only mode on macOS, iOS, and Android can prompt users to pick a certificate. Exclude those platforms from report-only device compliance rules if you don’t want unexpected end-user prompts during testing.

How Do Assignments Define Who, What, and Where a Policy Applies?#

Assignments are the conditional side of the if-then model. The Users assignment supports inclusions and exclusions across individual accounts, groups, directory roles, and guest or external user types. Always define both sides: who is in scope and who is explicitly excluded. Roles are particularly useful for targeting privileged access policies at Global Administrators, Security Administrators, and Exchange Administrators without building bespoke groups.

The Target resources assignment covers cloud apps, app filter groups, authentication context, and Global Secure Access. Authentication context lets you require step-up authentication for specific sensitive actions inside an app, such as accessing a high-sensitivity SharePoint site, without requiring stronger auth for the entire application.

Conditions refine assignment scope further. The key conditions are:

  • Device platforms: apply different controls to Windows, macOS, iOS, Android, and Linux.
  • Client apps: distinguish browser sessions from modern auth clients or legacy authentication protocols.
  • Filter for devices: use device attribute rules (for example, device.extensionAttribute1 -eq "Privileged") to target specific hardware classes.

Source: Microsoft Learn, “Conditional Access: Assignments”, retrieved 2026-05-22.

How Do Named Locations Strengthen Your Network Conditions?#

Named locations give Conditional Access a geographic and network context it would otherwise lack. Two location types are available: IP-based and Country-based. A tenant supports up to 195 named locations, each holding up to 2,000 IP ranges in CIDR notation (minimum prefix /8). IP-based locations can be marked as trusted, which improves Entra ID Protection’s risk scoring accuracy. A sign-in from a trusted location is less likely to be flagged as atypical. Source: Microsoft Learn, “Conditional Access: Network assignment”, retrieved 2026-05-22.

Country-based locations use IP geolocation data by default. Microsoft Authenticator can optionally supply GPS coordinates for the same evaluation, giving higher precision for mobile sign-ins.

VPN users are the most common source of named location confusion. The policy evaluates the VPN exit IP, not the user’s physical location. An employee working in Berlin through a UK exit node will match UK-based location policies. For organisations moving to a Zero Trust network model, Global Secure Access provides source IP restoration: the policy evaluates the user’s actual device IP rather than the VPN egress point.

One UI change to note: the assignment label was renamed from “Location” to “Network” in the Entra admin centre. Existing policies configured against the old label continue to work without changes, but new policy creation uses the updated label.

Which Grant Controls Are Available on E3 Today?#

Every grant control below is available with Entra ID P1, included in Microsoft 365 E3. You don’t need E5 to build a strong access control posture. The six primary grant controls cover the full range from hard block to conditional access based on device health and application management state. Source: Microsoft Learn, “Conditional Access: Grant controls”, retrieved 2026-05-22.

Block access is the simplest and most absolute control. When a Block policy applies, no other grant policy overrides it.

Require MFA triggers the user’s configured authentication method. Windows Hello for Business satisfies this control for compliant Windows devices.

Require authentication strength is a more precise version of “Require MFA.” Three built-in tiers are available:

TierAllowed Methods
MFA strengthPassword + Authenticator push, FIDO2 security key, Windows Hello for Business, Certificate-Based Auth (MFA), Temporary Access Pass, OATH tokens
Passwordless MFA strengthAuthenticator phone sign-in, FIDO2 security key, Windows Hello for Business, Certificate-Based Auth (MFA)
Phishing-resistant MFA strengthFIDO2 security key, Windows Hello for Business / platform credential, Certificate-Based Auth (MFA)

Custom authentication strengths are available when built-in tiers don’t match your method mix. One constraint: you cannot combine “Require MFA” and “Require authentication strength” in the same policy. Pick one. Source: Microsoft Learn, “Authentication strengths”, retrieved 2026-05-22.

Require device marked as compliant relies on Intune compliance policies. Supported platforms are Windows 10+, iOS, Android, macOS, and Linux Ubuntu. An Intune licence is required separately; it doesn’t come with Entra ID P1 alone.

Require Microsoft Entra hybrid joined device applies only to Windows devices joined to both on-premises Active Directory and Entra ID. iOS, Android, and macOS are excluded from this control.

Require app protection policy enforces Intune Mobile Application Management (MAM). iOS and Android are generally available. Windows is in preview. This control is the correct migration target for any policy that previously used “Require approved client app.”

Deprecation alert: “Require approved client app” was retired in March 2026. Any existing Conditional Access policy that uses this grant control as the only option must be migrated to “Require app protection policy.” If an OR combination of both controls exists in a policy, it continues to work while apps are updated. Three apps do not support app protection policy: Kaizala, Skype for Business, and Visio. For those apps, “Require approved client app” must remain where still applicable. Source: Microsoft Learn, “Conditional Access: Grant controls”, retrieved 2026-05-22.

Licence: All grant controls in this section (including authentication strengths) require Entra ID P1, included in Microsoft 365 E3, Microsoft 365 F1/F3, Microsoft 365 Business Premium, and EMS E3. Source: Microsoft Learn, “Microsoft Entra Licensing”, retrieved 2026-05-22.

What Session Controls Can You Apply with a P1 Licence?#

Session controls adjust what authenticated users can do after sign-in, without blocking access entirely. Four session controls are available on Entra ID P1. Source: Microsoft Learn, “Conditional Access: Session controls”, retrieved 2026-05-22.

Sign-in frequency sets how often a user must reauthenticate. You can require reauthentication on every access attempt (useful for privileged role sessions) or set a configurable interval in hours or days. The default persistent session behaviour in most browsers keeps users signed in far longer than security posture warrants for sensitive applications.

Persistent browser session controls whether the browser stores a persistent session cookie. Setting this to “Never persistent” forces reauthentication when the browser closes, which is appropriate for shared or unmanaged devices.

App-enforced restrictions delegate session control to SharePoint Online and Exchange Online directly. SharePoint uses this to enforce limited-access views, for example blocking downloads from unmanaged devices without needing Microsoft Defender for Cloud Apps.

Conditional Access App Control is covered in the next section. It appears as a session control toggle in the CA policy interface, but it requires a separate licence not included with Entra ID P1.

How Does Conditional Access App Control Work, and What Does It Cost?#

Conditional Access App Control uses a reverse proxy architecture to sit between the user’s browser and the target application. Non-Edge browsers access the app through a URL with the *.mcas.ms suffix appended. Microsoft Edge gets native in-browser protection without the URL modification. The proxy intercepts session activity and can apply granular session policies regardless of whether the app is managed by your organisation. Source: Microsoft Learn, “Conditional Access App Control”, retrieved 2026-05-22.

The feature works with SAML 2.0 and OIDC apps, including third-party SaaS applications onboarded to Microsoft Defender for Cloud Apps (MDCA). It also works with on-premises apps published through the Entra application proxy.

Session policy activities available through CAAC:

  • Prevent data exfiltration: block download, cut, copy, and print within the session.
  • Protect on download: apply Microsoft Purview sensitivity labels to downloaded files instead of blocking them entirely. If you haven’t yet configured those labels, see the guide on how to configure DLP policies in Microsoft 365.
  • Prevent upload of unlabeled files: stop users from uploading unclassified content to sanctioned apps.
  • Block potential malware uploads: scan uploads for known malware signatures before they reach the app.
  • Monitor sessions: log all session activity for compliance and investigation without enforcing any block.

Two limitations catch organisations by surprise. First, the Teams desktop application is NOT supported for CAAC. Browser sessions only. If your session policy targets Teams, unmanaged users on the desktop client fall outside the proxy entirely. Second, session policies have a 50 MB file size limit. Files above that threshold bypass session inspection entirely. Design download controls with this limit in mind when handling large engineering or finance files.

Licence: Conditional Access App Control requires Microsoft Defender for Cloud Apps. This is included in Microsoft 365 E5 Security, Microsoft 365 E5, and EMS E5. It is NOT included in Microsoft 365 E3 or standalone Entra ID P1/P2 licences. Source: Microsoft Learn, “Conditional Access App Control”, retrieved 2026-05-22.

What Does Upgrading to E5 Actually Unlock? Risk-Based Conditional Access#

Risk-based Conditional Access is the primary identity security upgrade that E5 unlocks. Entra ID Protection runs a continuous machine-learning analysis of every sign-in and user account in your tenant. It surfaces two risk signals: sign-in risk (the probability that a specific authentication attempt is not from the legitimate user) and user risk (the probability that a user account has been compromised). Both require Entra ID P2. Source: Microsoft Learn, “Risk-based Conditional Access policies”, retrieved 2026-05-22.

Sign-In Risk Policy#

Microsoft recommends targeting medium AND high sign-in risk levels. Low risk generates too much noise for a production enforcement policy. The recommended grant configuration pairs authentication strength (built-in MFA strength) with a session control of sign-in frequency: every time, forcing full reauthentication on each risky sign-in.

For passwordless deployments, use Passwordless MFA strength or Phishing-resistant MFA strength instead, so users who have already removed passwords aren’t prompted to use a method they don’t have.

One critical prerequisite: users who have not completed MFA registration and trigger a risky sign-in receive error AADSTS53004 and are completely blocked. They cannot self-remediate. Ensure organisation-wide MFA registration is complete before enabling sign-in risk enforcement. Source: Microsoft Learn, “Conditional Access policy: Sign-in risk”, retrieved 2026-05-22.

User Risk Policy#

Microsoft recommends targeting high user risk only. Medium user risk generates too many false positives for automated enforcement. The grant for user risk is “Require risk remediation”, which automatically applies authentication strength and sets sign-in frequency to every time.

User risk enables Self-Service Password Reset (SSPR) self-remediation. The user resets their password, Entra ID Protection reassesses the risk level, and the account is unblocked without helpdesk involvement. This is the operational efficiency argument for P2 beyond the security posture improvement. Source: Microsoft Learn, “Conditional Access policy: User risk”, retrieved 2026-05-22.

Exclude service accounts and break-glass accounts from both risk policies. Service accounts typically exhibit atypical sign-in patterns by design, and risk policies will generate constant alerts for them.

The real upgrade value is continuous, automated response. Without P2, your process for responding to a compromised credential is: detection, alert, analyst review, ticket opened, helpdesk calls the user. With P2, the policy triggers MFA remediation or SSPR the moment the risk fires. The attack window shrinks from hours to seconds.

Licence: Sign-in risk and user risk conditions in Conditional Access require Entra ID P2, included in Microsoft 365 E5, Microsoft Defender Suite, and EMS E5. Entra ID P2 is also available as an add-on for Microsoft 365 E3. Source: Microsoft Learn, “Microsoft Entra Licensing”, retrieved 2026-05-22.

How Do You Roll Out Conditional Access Policies Safely?#

A phased rollout is not optional. CA policies are additive, and any applicable Block policy wins regardless of the order policies are evaluated. Rushing to enforcement before validating report-only results is the most common cause of lockout incidents. The three-phase model below applies to any new policy, not just a full deployment. Source: Microsoft Learn, “Plan a Conditional Access deployment”, retrieved 2026-05-22.

PhaseTimelinePoliciesMinimum Report-Only Period
Phase 1Weeks 1-2Block legacy authentication, secure MFA registration, phishing-resistant MFA for privileged roles1 week
Phase 2Weeks 2-3Strong authentication for all users and guests, app protection policy for mobile, require MFA for device join/registration1 week
Phase 3Weeks 3-4Sign-in risk policy, user risk policy, token protection, device code flow restriction (P2 required)1 week

Phase 1 is the highest-value starting point. Legacy authentication protocols (Basic Auth, IMAP, POP3, SMTP Auth) bypass modern CA entirely. Securing the MFA registration page closes the registration loophole: without it, an attacker with a stolen password can register their own MFA device before the legitimate user notices. Use FIDO2 or Windows Hello for Business for privileged roles, not Authenticator push.

Phase 2 extends strong authentication to the broader user population. App protection policy for iOS and Android closes the unmanaged mobile access gap without requiring device enrolment in Intune.

Phase 3 requires Entra ID P2. Token protection and device code flow restriction reduce token theft and phishing-resistant bypass techniques. Every phase starts in report-only. No exceptions.

Frequently Asked Questions#

What is the difference between report-only mode and disabled in Conditional Access?#

A disabled policy is completely inactive and logs nothing. Report-only mode runs full policy evaluation and logs the outcome for every matching sign-in, but does not enforce any grant or session control. The four logged outcome states are: Success, Failure, User action required, and Not applied. Microsoft recommends at least one week in report-only before enforcement for any new policy. Source: Microsoft Learn, “Report-only mode”, retrieved 2026-05-22.

Can Microsoft 365 E3 users configure risk-based Conditional Access?#

No. Sign-in risk and user risk conditions require Entra ID P2. E3 includes P1 only. E3 organisations can purchase the Entra P2 add-on per user to unlock risk-based CA without upgrading to E5. Alternatively, a full upgrade to Microsoft 365 E5 includes P2 across the board. Source: Microsoft Learn, “Microsoft Entra Licensing”, retrieved 2026-05-22.

What should I do with existing “Require approved client app” policies?#

Migrate them to “Require app protection policy” before the retirement breaks them. The safest path: add “Require app protection policy” as an OR option alongside the existing grant, test in report-only, then remove the deprecated grant once validation is complete. Three exceptions remain: Kaizala, Skype for Business, and Visio must continue using “Require approved client app” where still applicable.

Why are users getting AADSTS53004 errors after enabling sign-in risk policies?#

AADSTS53004 means the user has not completed MFA registration but triggered a risky sign-in. Risk policies require MFA remediation, which is impossible without a registered MFA method. Complete MFA registration for all users before enabling sign-in risk enforcement. Run the Authentication Methods Activity report in the Entra admin centre to identify unregistered users first. Source: Microsoft Learn, “Conditional Access: Sign-in risk”, retrieved 2026-05-22.

Does Conditional Access App Control work with the Microsoft Teams desktop app?#

No. The Teams desktop application is not supported for CA App Control. CAAC applies only to browser sessions. Users on the Teams desktop client bypass the proxy entirely. Users accessing Teams through a browser on an unmanaged device are subject to CAAC session policies. Design your unmanaged device policies with this boundary in mind. Source: Microsoft Learn, “CA App Control”, retrieved 2026-05-22.

Start in Report-Only Mode, Then Build From There#

Conditional Access is not a deploy-once configuration. It grows with your organisation’s licence posture, device management maturity, and risk tolerance. Start by reviewing any existing “Require approved client app” policies, since those need migration now. Audit your break-glass accounts and verify they’re excluded from every policy. Then run your first new policy in report-only for at least a week before enforcing.

For E5 evaluation: activate an Entra ID P2 trial and enable the sign-in risk policy in report-only mode for two weeks. The Entra ID Protection risky sign-ins report will show you real risk events from your tenant. That live data is far more persuasive for an upgrade business case than any vendor comparison sheet.

The Zero Trust identity layer starts with P1 and the policies you can build today. P2 adds the automated risk response layer on top. Both are worth building deliberately rather than fast.
