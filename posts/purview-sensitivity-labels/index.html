How to Configure Microsoft Purview Sensitivity Labels and Label Policies

Step-by-step guide to Microsoft Purview sensitivity labels: plan a label taxonomy, create labels and sublabels, publish policies, configure auto-labeling, and monitor activity. E3 vs E5 licensing explained.
May 24, 2026 · 17 min · Alexander Schirmer

In 2025, the average global cost of a data breach reached $4.44 million, and breaches traced to malicious insiders cost even more — $4.92 million on average — according to the IBM Cost of a Data Breach Report 2025. The question organizations repeatedly fail to ask before a breach is: do we even know where our sensitive data lives? Microsoft Purview sensitivity labels are the answer to that question. They classify data at the file and email level, apply persistent protection that travels with the document, and enforce usage rights regardless of where the file ends up. This guide walks through every step: taxonomy planning, label creation, label policies, auto-labeling, monitoring, and the PowerShell cmdlets that make it repeatable.

Key Takeaways

  • Sensitivity labels classify data at the file and email level; protection persists wherever the file travels, unlike DLP policies that only act at the network boundary
  • Start with three parent labels — Public, Internal, Confidential — and add sublabels for specific handling requirements before creating anything in the portal
  • Manual labeling in Office apps is available from Microsoft 365 E3; auto-labeling policies (service-side, SharePoint and Exchange) require E5 or the M365 Information Protection and Governance add-on
  • Publishing a label policy takes under ten minutes; the labels appear in Office apps within 24 hours for most tenants
  • PowerShell via Connect-IPPSSession lets you list, create, and publish labels and policies without touching the portal — essential for multi-tenant deployments

What Are Sensitivity Labels and How Are They Different from DLP?#

Sensitivity labels classify a document or email by attaching a metadata tag directly to the file. That tag persists — it survives download, forwarding, and attachment to a new email. DLP policies, by contrast, inspect content at the point of transmission and decide whether to allow or block an action. DLP is a network-level enforcement mechanism. Sensitivity labels are a classification mechanism that can trigger DLP conditions, encryption, and usage restrictions all at once.

The practical difference matters for data that leaves your perimeter. A DLP policy can block a user from emailing a file containing a credit card number to an external recipient. A sensitivity label applied to that same file encrypts it so that even if DLP is bypassed — forwarded from a personal device, downloaded to a USB drive, uploaded to a shadow IT service — only users with the correct Azure AD permissions can open it. The two controls work together: your DLP policies can use sensitivity labels as conditions, creating a layered enforcement model where classification drives both the DLP rule and the encryption.

In 2025, the Varonis Data Risk Report found that 99% of organizations had sensitive data exposed to AI tools including Microsoft Copilot, and that on average, every employee had access to roughly 25,000 sensitive folders on day one. Sensitivity labels with encryption are the only control that restricts what Copilot can retrieve from labeled content. Without labels, Copilot returns content based on permissions alone.

How Do You Plan a Sensitivity Label Taxonomy?#

Your label taxonomy determines what labels appear in Office apps and what protection each label applies. Getting this right before opening the portal saves weeks of remediation. A taxonomy designed by committee after labels are already published causes confusion and inconsistent use.

The recommended starting point for most Microsoft 365 tenants is a three-tier hierarchy:

Parent LabelSublabelClassification Intent
Public(no sublabels)Content approved for public consumption — press releases, marketing materials, published documentation
InternalGeneralGeneral internal use — meeting notes, internal reports, team communications
InternalRestrictedInternal content with a limited audience — HR discussions, budget drafts
ConfidentialAll EmployeesSensitive but broadly available internally — policies, strategy decks
ConfidentialRecipients OnlyEncrypted and restricted to named recipients — legal documents, board materials
ConfidentialHighly ConfidentialHighest protection — mergers, regulatory submissions, source code

Microsoft recommends no more than five parent labels and no more than five sublabels per group, to keep the picker usable for end users. More labels mean more friction, which means users default to the lowest available option or skip labeling entirely. Start small and add labels only when a documented business requirement drives the need.

Naming matters. Labels with names that describe the impact (“Recipients Only” rather than “Label-3”) are adopted more consistently. Each label needs a tooltip — a short explanation of when to apply it — that appears when users hover over the label in Office apps. Write tooltips in plain language that reflects how the user thinks about the content, not how IT classifies it.

How Do You Create Sensitivity Labels in the Microsoft Purview Portal?#

Labels are created in the Microsoft Purview compliance portal at purview.microsoft.com. Navigate to Information protection in the left menu, then select Labels. You need the Sensitivity Label Administrator or Compliance Administrator role to create labels. The Security Reader role gives read-only access.

Creating a Parent Label#

Select Create a label. On the first screen, enter:

  • Name (internal identifier — cannot be changed after creation)
  • Display name (what users see — can be changed)
  • Description for users (tooltip — plain language, max 1,000 characters)
  • Description for admins (internal notes visible only in the portal)

On the Scope screen, select the scopes this label applies to. The default in 2026 is Files and other data assets (which covers Office files, PDFs, Loop components, Power BI, and Fabric). Add Emails if the label should be available in Outlook. Meetings requires both Files and Emails to be selected first. Groups and sites applies the label to Microsoft 365 groups, Teams sites, and SharePoint sites — configure this separately.

For a parent label with sublabels, you can leave protection settings empty on the parent. The parent becomes a grouping header and the sublabels carry the actual protection.

Creating Sublabels#

After saving the parent, return to the Labels list. Select the parent label and choose Add sublabel. The sublabel configuration follows the same steps. On the protection settings screen, configure the protection specific to that sublabel — for example, a sublabel for “Recipients Only” applies encryption with restricted permissions.

The order of labels in the list determines the order in the Office picker. Drag labels to set the priority. Lower priority labels should appear at the top of the list, so that the least protective option is the first thing users see (and less likely to be accidentally chosen for sensitive content). This is counterintuitive but follows Microsoft’s recommended configuration.

What Protection Settings Can You Configure on a Label?#

Each label supports three independent protection mechanisms: encryption, content marking, and auto-labeling conditions (covered in the auto-labeling section below).

Encryption#

Encryption is the most powerful protection setting. When enabled, it restricts who can open, edit, print, copy, or forward the content regardless of where the file is stored. Two modes are available:

Assign permissions now — you define the usage rights at label creation time. Select specific users or Azure AD groups and assign a permission level: Co-Owner (full control), Co-Author (view, edit, print, copy, forward), Reviewer (view and edit), Viewer (view only), or Custom (select individual rights from the complete list). This is the right choice for labels with a known audience — for example, a board-materials label scoped to the Board Members group.

Let users assign permissions — users choose recipients and permission levels when they apply the label. This is appropriate for ad-hoc confidential sharing where the sender, not IT, determines the audience. The label enforces that encryption is required; the sender determines who can access the content.

For labels used internally across the organization, the simplest setup is Assign permissions now → All users and groups → Co-Author. This encrypts content so only your Azure AD tenant can open it, while preserving full editing rights for all employees.

Content Marking#

Content marking adds visible indicators to files and emails: watermarks (text overlaid on the document page), headers (text at the top of each page), and footers (text at the bottom). These are separate from encryption and do not restrict access.

Watermarks support variable substitution. The token ${LabelName} inserts the current label’s display name, ${Item.SensitivityLabel} inserts the label name, and ${Item.Label} inserts additional label metadata. Set the font size to 48pt or higher for watermarks to remain legible on printed documents.

Content marking is most useful for Confidential and higher labels, where a visible indicator helps users recognize sensitive content and reminds them to handle it appropriately. Do not apply watermarks to Internal or Public labels — it adds visual noise without security value.

How Do You Publish a Sensitivity Label Policy?#

Creating labels doesn’t make them visible in Office apps. You publish labels to users via a label policy. A label policy specifies which labels are available, which users or groups can see them, and behavioral settings like the default label and mandatory labeling.

Navigate to Information protection → Label policies → Create policy. On the labels screen, select the labels to include in this policy. You can create multiple policies with different label sets — for example, a base policy that publishes Public, Internal, and Confidential to all users, and a separate policy that also publishes Highly Confidential to the legal team.

Policy Settings#

After selecting labels, configure the behavioral settings:

Default label — select a label that is applied automatically when no label has been set. Most organizations set this to “Internal / General” for files. A default label does not apply encryption on its own; it sets a starting classification that users can then change.

Mandatory labeling — when enabled, users cannot save a file or send an email without selecting a label. This eliminates the “no label” state entirely. Enable this for new deployments only after a training campaign — mandatory labeling without training causes support tickets. Consider enabling it on email before documents, as email labeling is more intuitive for most users.

Downgrade justification — when enabled, users must provide a reason when they change a label to a lower classification. Justification text is logged in the audit trail and visible in Activity Explorer. This creates accountability without preventing legitimate downgrades.

Policy scope — publish the policy to specific users or groups rather than all users if you’re rolling out in phases. A phased rollout — pilot group first, then department-by-department — reduces the volume of confusion tickets and lets you tune the default label before broad deployment.

After configuration, the policy propagates to users within 24 hours for most tenants. Labels appear in the Sensitivity dropdown in Office apps (Word, Excel, PowerPoint, Outlook) and in the Purview Information Protection bar if the built-in client is active.

What Is Auto-Labeling and How Does It Work?#

Auto-labeling reduces the burden on users by applying labels automatically based on content inspection. Two distinct mechanisms exist: client-side auto-labeling and service-side auto-labeling policies.

Client-Side Auto-Labeling#

Client-side auto-labeling runs inside Office apps when a user is editing a document or composing an email. It inspects the content in real time and either recommends a label (a notification bar appears with a suggestion) or applies one automatically. This runs on the user’s device and does not require E5 — it’s part of the built-in labeling client in Microsoft 365 Apps.

Configure client-side conditions on individual labels under the Auto-labeling for files and emails tab within the label configuration. Set conditions using sensitive information types (SITs), trainable classifiers, or sensitivity labels already applied to the file. For example: if the document contains three or more UK National Insurance numbers, recommend the “Confidential / Recipients Only” label.

Use Recommend mode before switching to Automatically apply. Recommendation surfaces the label suggestion to the user with a one-click accept, giving you a period to observe which content is being matched and whether false positives are occurring before making the application automatic.

Service-Side Auto-Labeling Policies#

Service-side auto-labeling policies run in the cloud against content already at rest in SharePoint, OneDrive, and Exchange — content that users haven’t touched. This requires Microsoft 365 E5 or the M365 Information Protection and Governance add-on. Without E5, service-side policies are not available.

Configure service-side policies under Information protection → Auto-labeling policies. You specify the locations to scan (specific SharePoint sites, OneDrive accounts, or Exchange mailboxes), the conditions that trigger labeling (SITs or trainable classifiers), and the label to apply.

Run the policy in Simulation mode first. Simulation scans the selected locations and reports which items would be labeled, without applying labels. Review the simulation results in the policy dashboard, adjust conditions to reduce false positives, then switch to enforcement. This is the same phased approach used in DLP simulation mode — the principle carries across both features.

How Do You Monitor Sensitivity Labeling Activity?#

Two tools provide visibility into how labels are being used: Content Explorer and Activity Explorer. Both are in the Microsoft Purview compliance portal under Data classification.

Content Explorer shows a snapshot of labeled content across Exchange, SharePoint, and OneDrive. Filter by label, SIT, or file type to see where your sensitive data lives. The view is point-in-time, not real-time. It’s most useful during the first 30 days of a deployment to understand the distribution of labeled content and identify locations that haven’t yet been covered.

Activity Explorer shows a timeline of labeling events: when a label was applied, changed, removed, or when a file with a label was downloaded, printed, or accessed. Filter by user, label, location, and activity type. Enable downgrade justification in your label policy and those justification texts appear here too, giving you a searchable audit trail for classification decisions.

Activity Explorer data is retained for 30 days by default. If you need longer retention for compliance purposes, connect Purview audit logs to Microsoft Sentinel or export via the Office 365 Management API.

For Conditional Access integration, sensitivity labels on files can be used as conditions in Conditional Access policies via app-enforced restrictions in SharePoint and OneDrive. This lets you require compliant devices to access Confidential-labeled SharePoint sites — identity and data controls working together.

E5 or the Information Protection and Governance add-on unlocks service-side auto-labeling policies — the mechanism that retroactively labels existing content at rest without user interaction.

PowerShell: Managing Sensitivity Labels with the Security and Compliance Module#

The Purview sensitivity label cmdlets run in Security & Compliance PowerShell, not Exchange Online PowerShell. Connect using Connect-IPPSSession from the ExchangeOnlineManagement v3 module.

#Requires -Modules ExchangeOnlineManagement

Connect-IPPSSession -ShowBanner:$false

List All Sensitivity Labels#

Get-Label | Select-Object -Property DisplayName, Name, Priority, ContentType, Disabled

This returns all labels in the tenant, including sublabels. The Priority property reflects the label order in the Purview portal — higher priority numbers appear lower in the picker. Use ContentType to confirm which scopes the label covers (File, Email, Site, UnifiedGroup, etc.).

Create a New Label#

The example below creates a “Confidential — Recipients Only” label with encryption assigned to specific users and a footer content marking.

#Requires -Modules ExchangeOnlineManagement

Connect-IPPSSession -ShowBanner:$false

$labelParams = @{
    DisplayName              = 'Confidential - Recipients Only'
    Name                     = 'Confidential_RecipientsOnly'
    Tooltip                  = 'Use for documents shared with named external recipients only. Encryption restricts access to specified users.'
    ParentId                 = 'Confidential'
    ContentType              = 'File, Email'
    EncryptionEnabled        = $true
    EncryptionProtectionType = 'UserDefined'
    ApplyContentMarkingFooterEnabled = $true
    ApplyContentMarkingFooterText    = 'CONFIDENTIAL — RECIPIENTS ONLY'
    ApplyContentMarkingFooterFontSize = 10
    ApplyContentMarkingFooterAlignment = 'Center'
    ErrorAction              = 'Stop'
}
New-Label @labelParams

The EncryptionProtectionType = 'UserDefined' setting enables the “Let users assign permissions” mode — the sender specifies recipients when applying the label. Replace with EncryptionProtectionType = 'Template' and add -EncryptionRightsDefinitions to assign permissions at label creation time for a fixed audience.

Create and Publish a Label Policy#

#Requires -Modules ExchangeOnlineManagement

Connect-IPPSSession -ShowBanner:$false

$policyParams = @{
    Name                     = 'M365-Sensitivity-Baseline'
    Labels                   = @(
        'Public',
        'Internal_General',
        'Internal_Restricted',
        'Confidential_AllEmployees',
        'Confidential_RecipientsOnly',
        'Confidential_HighlyConfidential'
    )
    Comment                  = 'Baseline sensitivity label policy — all M365 users'
    ExchangeLocation         = 'All'
    SharePointLocation       = 'All'
    OneDriveLocation         = 'All'
    ErrorAction              = 'Stop'
}
New-LabelPolicy @policyParams

After running New-LabelPolicy, the labels propagate to users within 24 hours. To target the policy to a specific group instead of all users, replace ExchangeLocation = 'All' with the group’s SMTP address.

To verify the policy is correctly configured, run:

Get-LabelPolicy -Identity 'M365-Sensitivity-Baseline' | Select-Object -Property Name, Labels, ExchangeLocation

Frequently Asked Questions#

What is the difference between sensitivity labels and Azure Information Protection?#

Azure Information Protection (AIP) was the predecessor to Microsoft Purview sensitivity labels. The AIP classic client and the AIP unified labeling client have both been replaced by built-in labeling in Microsoft 365 Apps. If your organization uses the AIP classic client, Microsoft recommends migrating to built-in labeling. The underlying label infrastructure (unified labeling store) is the same — only the client-side application changed. (Microsoft Learn, 2025)

Can sensitivity labels encrypt files that are stored on-premises?#

Sensitivity labels with encryption can be applied to files stored on-premises via the Microsoft Purview Information Protection scanner, which scans file shares and SharePoint Server. Once applied, the encryption persists regardless of where the file is stored. The scanner requires AIP Plan 1 or Plan 2 licensing. Encrypted files stored on-premises can only be decrypted by users who have Azure AD-based access rights — the encryption check requires internet connectivity to validate usage rights.

How long does it take for a new label policy to reach users?#

New and modified label policies propagate to users within 24 hours in most tenants. Outlook on the web typically reflects changes faster than the Outlook desktop client. If a policy change is urgent, users can force a policy refresh by closing and reopening their Office apps or signing out and back in. The Get-LabelPolicy cmdlet shows the current configuration but does not show propagation status to individual endpoints.

Does Microsoft 365 E3 include auto-labeling?#

E3 includes client-side auto-labeling in recommendation mode — Office apps can suggest a label based on content detection, and the user accepts or changes it. Automatic application without user interaction (also client-side) requires EMS E5 or AIP Plan 2. Service-side auto-labeling policies that scan and label existing SharePoint and Exchange content at rest require Microsoft 365 E5 or the M365 Information Protection and Governance add-on. (Microsoft Purview service description, 2025)

Can Copilot access content protected by sensitivity labels?#

Microsoft 365 Copilot respects sensitivity label encryption. If a document has an encryption label that does not include the EXTRACT usage right, Copilot cannot retrieve or summarize that content. By default, the Co-Owner and Co-Author permission levels include the EXTRACT right, meaning Copilot can access content encrypted for internal use. The Viewer permission level does not include EXTRACT — labeling documents with Viewer-only encryption prevents Copilot from referencing them in generated responses. (Microsoft Learn, 2025)

Do sensitivity labels apply to SharePoint sites and Teams?#

Yes, but this requires the Groups and sites scope, which is not enabled by default. To enable it, you must run Execute-AzureAdLabelSync in Security & Compliance PowerShell and wait up to 24 hours for the setting to propagate. Once enabled, you can apply labels to Microsoft 365 Groups, Teams channels, and SharePoint sites. The site label controls guest access settings, external sharing settings, and whether unmanaged devices can access the site — separate from the file-level labels applied to documents within the site. (Microsoft Learn, 2025)

Sensitivity labels give you a persistent classification layer that DLP policies can reference and encryption can enforce. The taxonomy planning step — before anything touches the portal — determines whether the program succeeds or produces a graveyard of unused labels. Start with three parent labels, get mandatory labeling in place on email, monitor via Activity Explorer for 30 days, then evaluate whether service-side auto-labeling policies are worth the E5 investment based on what you observe in Content Explorer.
