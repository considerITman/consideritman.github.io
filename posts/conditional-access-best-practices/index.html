Conditional Access Best Practices in Microsoft 365 (2025 Deployment Guide)

Conditional Access deployment guide for M365: block legacy auth, enforce phishing-resistant MFA, add E5 risk policies. MFA stops 99.9% of compromised sign-ins.
May 22, 2026 · 15 min · Alexander Schirmer

In the first half of 2025, identity-based attacks against Microsoft 365 tenants surged 32%, according to the Microsoft Digital Defense Report 2025. Microsoft now processes over 600 million identity attacks every single day. Conditional Access is the policy engine designed to stop them — but only if it’s configured correctly.

Most organisations have Conditional Access available and haven’t deployed it. Others deployed it years ago and haven’t reviewed it since. Both are dangerous. This guide doesn’t explain what Conditional Access is. For that, see our complete guide to Conditional Access policies in Entra ID. This guide tells you what to actually build: which policies, in which order, with which settings, and how to automate the deployment with Microsoft Graph PowerShell.

Key Takeaways

  • In 2025, 99.9% of compromised M365 accounts had no MFA (Microsoft Partner Center, 2025) — CA policies are the fix.
  • M365 E3 (Entra ID P1) covers the full CA policy engine; E5 (Entra ID P2) adds risk-based automated response.
  • A 2026 enforcement change applies “All resources” CA policies to OIDC flows regardless of exclusions — audit your tenant now.

Why Can’t Conditional Access Wait in 2025?#

In 2025, more than 99.9% of compromised Microsoft 365 accounts had no MFA enabled, according to Microsoft Learn’s Partner Center security statistics, retrieved 2026-05-22. That figure has been consistent for years. The attack methods evolve — password spray, phishing, adversary-in-the-middle token theft — but the underlying gap stays the same. Unprotected accounts are the path of least resistance, and attackers know it.

There’s also a 2026 action item most guides don’t mention. Microsoft is rolling out an enforcement change between March and June 2026: Conditional Access policies targeting “All resources” will now be enforced for OIDC-scope sign-ins even when resource exclusions are present. If your tenant has a policy that uses “All resources” with specific app exclusions, users in affected sign-in flows will start receiving unexpected MFA or device compliance prompts. Check those policies now before the rollout hits your tenant. Source: Microsoft Learn, “Common security policies for Microsoft 365”, retrieved 2026-05-22.

And if you have “Require approved client app” in any policy, it’s already retired as of March 2026. The replacement is “Require app protection policy,” which provides the same data loss protection with broader device coverage — including unenrolled devices.

According to the Microsoft-managed Conditional Access policies documentation, retrieved 2026-05-22, MFA continues to reduce the risk of account compromise by more than 99%. That’s not a number you ignore. The question isn’t whether to deploy Conditional Access. It’s how to do it without locking out your users in the first week.

What Does Microsoft 365 E3 Give You for Conditional Access?#

In 2025, a frequent misconception among IT teams is that Conditional Access is an “E5 feature.” It isn’t. Microsoft 365 E3 includes Entra ID P1, which covers the complete CA policy engine: named locations, authentication strength, device compliance grants, app protection controls, and the ability to block legacy authentication outright. Source: Microsoft Learn, Entra ID licensing, retrieved 2026-05-22.

What E5 adds is automated risk response. Entra ID P2 — included in Microsoft 365 E5 and the Microsoft Defender Suite — enables risk-based Conditional Access: policies that evaluate real-time sign-in risk and user risk signals from Entra ID Protection and respond automatically. You can’t replicate that with E3 policies and manual reviews. It’s a fundamentally different capability.

The practical decision is this: don’t wait for an E5 upgrade to start deploying CA. E3 blocks the majority of real-world attacks. E5 adds the automated, continuous response layer for high-value targets and regulated environments. Build your P1 foundation first, then layer P2 on top if you have the licence.

According to Microsoft Learn’s Entra ID licensing documentation, retrieved 2026-05-22, Microsoft Entra ID P1 is included in Microsoft 365 E3, F1, F3, and Business Premium. Entra ID P2 — which adds risk-based Conditional Access via ID Protection — is included in Microsoft 365 E5 and the Microsoft Defender Suite. The capability difference isn’t marginal: risk-based policies respond to signals that no manual policy can replicate.

How Should You Phase a Conditional Access Rollout?#

Deploying CA policies all at once is how you end up rolling everything back on day two. Microsoft’s own deployment guide — Plan a Conditional Access deployment, retrieved 2026-05-22 — recommends a phased approach across four weeks. Each phase builds on the last. You don’t move to Phase 2 until Phase 1 has run in report-only mode for at least a week and you’ve verified the sign-in logs.

Before you touch Phase 1, two prerequisites must be in place. First: create two or more break-glass accounts — cloud-only Global Administrator accounts excluded from every CA policy by object ID, not by group. These are your recovery path if a misconfigured policy locks everyone out. Second: confirm every user you’re targeting has registered an MFA method. A policy that enforces MFA against users who haven’t registered yet blocks them completely.

Phase 1 — Foundation (Week 1–2): Three policies, all Entra ID P1. Block legacy authentication across all users and all apps. Secure the MFA registration page so users can only register from a trusted location (preventing attackers from registering their own MFA methods on a compromised account). Require phishing-resistant MFA — specifically FIDO2 security keys or Windows Hello for Business via authentication strength — for privileged directory roles.

Phase 2 — Core Authentication (Week 2–3): Four policies, all Entra ID P1. Require MFA for all users targeting all resources with no app exclusions. Extend MFA to guest accounts. Require app protection policy for iOS and Android. Require MFA for device join and device registration. By the end of Phase 2, every interactive sign-in from every user is covered.

Phase 3 — Advanced Protection (Week 3–4): This phase requires Entra ID P2. Block high-risk sign-ins. Require MFA or password change for high-risk users. Add token protection, device code flow restriction, and authentication transfer blocking. If you’re on E3, document this phase as a known capability gap and revisit when you add E5 or the E5 Security add-on.

Which Conditional Access Policies Does Every Tenant Need?#

Every Microsoft 365 tenant with Entra ID P1 should have these six policies deployed and enforced. This isn’t a “nice to have” list — it’s the minimum posture. The CyberCX 2025 analysis by Chris Chambers and Alan Johnstone, retrieved 2026-05-22, found that most tenants they assessed had fewer than four of these in place, with stale exclusions undermining the ones that were deployed.

CA01 — Block Legacy Authentication. Scope: all users, all apps, client app types exchangeActiveSync and other. Grant: Block. Always start here. Legacy authentication protocols can’t satisfy MFA challenges — attackers use them specifically to bypass CA. This policy alone closes one of the widest attack surfaces in any M365 environment.

CA02 — Require MFA for All Users. Scope: all users, all resources. Grant: require authentication strength (multifactor). No app exclusions. Microsoft explicitly recommends against app exclusions in a baseline MFA policy because apps like Teams and Exchange depend on multiple backend services — exclusions create unpredictable behaviour. Exclude your break-glass accounts by object ID.

CA03 — Phishing-Resistant MFA for Privileged Roles. Scope: Global Admins, Security Admins, Exchange Admins (and others with high-privilege roles), all resources. Grant: require authentication strength set to “Phishing-resistant” — FIDO2 security keys or Windows Hello for Business only. Push notifications and SMS don’t qualify. Admin accounts are the highest-value phishing targets.

CA04 — Secure MFA Registration. Scope: all users, user action “Register security information.” Conditions: require network location outside named trusted locations. Grant: require MFA. This prevents an attacker who has captured a password from registering their own MFA device while no MFA exists yet.

CA05 — Require App Protection Policy on Mobile. Scope: all users, Exchange Online and SharePoint, platforms iOS and Android. Grant: require app protection policy. This replaces the retired “Require approved client app.” Intune MAM is required but works on both enrolled and unenrolled devices.

CA06 — Block High-Risk Sign-ins (E5/P2 only). Scope: all users, all resources, sign-in risk: high. Grant: Block. If you’re on E3, document this slot in your policy register as a known gap. The risk-based signals from ID Protection are the automated layer that catches credential stuffing, impossible travel, and adversary-in-the-middle attacks in real time.

One limit worth keeping in mind: tenants have a maximum of 240 CA policies across all states (enabled, disabled, and report-only). Report-only policies count. Don’t accumulate stale test policies — clean them up after each validation cycle. Source: Microsoft Learn, Plan a Conditional Access deployment, retrieved 2026-05-22.

Automating Policy Creation with Microsoft Graph PowerShell#

Hand-building six (or sixteen) CA policies through the Entra admin centre invites human error and makes rollback painful. Microsoft Graph PowerShell — specifically New-MgIdentityConditionalAccessPolicy from the Microsoft.Graph.Identity.SignIns module — is the correct tool for repeatable, auditable policy deployment. Source: Microsoft Learn, New-MgIdentityConditionalAccessPolicy, retrieved 2026-05-22.

The required permission scopes are Policy.Read.All and Policy.ReadWrite.ConditionalAccess. You’ll need at least the Conditional Access Administrator role.

The example below creates CA01 — Block Legacy Authentication in report-only mode. Change state to "enabled" only after you’ve reviewed a week of sign-in logs and confirmed no legitimate services use legacy auth.

Import-Module Microsoft.Graph.Identity.SignIns

Connect-MgGraph -Scopes "Policy.Read.All", "Policy.ReadWrite.ConditionalAccess"

# Replace with your break-glass account object IDs
$breakGlassIds = @("<break-glass-account-1-object-id>", "<break-glass-account-2-object-id>")

$params = @{
    displayName = "CA01 - Block Legacy Authentication"
    state       = "enabledForReportingButNotEnforced"  # switch to "enabled" after report-only validation
    conditions  = @{
        clientAppTypes = @("exchangeActiveSync", "other")  # targets legacy auth clients
        applications   = @{
            includeApplications = @("All")
        }
        users = @{
            includeUsers = @("All")
            excludeUsers = $breakGlassIds
        }
    }
    grantControls = @{
        operator        = "OR"
        builtInControls = @("block")
    }
}

New-MgIdentityConditionalAccessPolicy -BodyParameter $params
Write-Host "CA01 created in report-only mode. Review sign-in logs for 7 days before enforcing."

exchangeActiveSync covers Exchange ActiveSync clients. other covers older SMTP, IMAP, POP3, and basic auth clients. Together, these two client app types define the full legacy auth surface. Omitting either leaves a gap.

Automating at scale. If you manage multiple tenants or want to deploy all six policies as a baseline, structure each policy as a hashtable in an array and loop with New-MgIdentityConditionalAccessPolicy. Store your break-glass object IDs and group IDs in a configuration object at the top, not hardcoded throughout the script.

What If API. As of July 2025, Microsoft released a Conditional Access “What If” API that allows programmatic impact testing before rollout — you pass a simulated sign-in payload and get back a policy evaluation result. This is the equivalent of the portal’s What If tool, callable from your CI/CD pipeline. Source: Microsoft Learn, CA Optimization Agent, retrieved 2026-05-22.

CA Optimization Agent. Released at Microsoft Ignite in November 2025, the Conditional Access Optimization Agent continuously monitors your tenant’s CA policies, flags unprotected users and apps, and surfaces suggested changes. After November 17, 2025, it runs under a dedicated Entra Agent ID rather than the activating user’s identity. It doesn’t replace manual policy design, but it closes the monitoring gap that most teams ignore. [INTERNAL-LINK: CA Optimization Agent setup → Microsoft Entra admin center documentation]

Five Mistakes That Leave Gaps in Your Coverage#

The difference between a secure CA deployment and a false sense of security often comes down to five configuration patterns. Chris Chambers and Alan Johnstone from CyberCX documented these from real engagements in their 2025 Conditional Access analysis, retrieved 2026-05-22.

1. Exclusion drift. Every CA policy accumulates exclusions over time — service accounts, VIP users, testing groups. Most of those exclusions were meant to be temporary. Attackers specifically target excluded accounts because they know those accounts bypass CA entirely. Review your exclusion lists quarterly. Any account excluded from your MFA policy needs a documented and active reason.

2. Browser sessions missed. It’s common to see device compliance or app protection controls applied to mobile and desktop clients but not to browser sessions of the same application. A user can open Exchange Online in Chrome on a personal, unmanaged laptop and bypass the controls entirely if the browser session isn’t in scope. Always check client app type coverage in your conditions.

3. Device code flow unblocked. Device code flow — the authentication method used by devices without a browser — is rarely needed in most tenants. It’s also one of the primary vectors for token theft attacks. Block it in Phase 3 (Entra ID P1 required, despite being in the advanced phase). The only users who need it are those using Azure CLI or similar tooling from headless devices, and that can be scoped with a narrow allow-policy.

4. Risk signals ignored. Many E5 tenants have Entra ID Protection active but haven’t wired the sign-in risk and user risk signals into their CA policies. The risk signals fire — the logs show them — but nothing enforces a response. If you have P2 licences, connect the signals. A high-risk sign-in that generates a detection and then proceeds to full access isn’t a security posture improvement.

5. No monitoring cadence. Conditional Access is not set-and-forget. The Entra ID Insights and Reporting workbook tracks the percentage of sign-ins in scope for CA policy, the percentage protected by MFA, and policy-level success and failure rates. Without Azure Monitor integration, this data stays in the sign-in logs but doesn’t aggregate. Build a monthly review of these metrics into your security operations cadence.

Frequently Asked Questions#

Does Conditional Access work with Microsoft 365 E3?#

Yes. Microsoft 365 E3 includes Entra ID P1, which gives you the full CA policy engine: named locations, authentication strength (including phishing-resistant MFA), device compliance, app protection policy, and legacy auth blocking. What E3 doesn’t include is risk-based CA — sign-in risk and user risk policies require Entra ID P2 (included in M365 E5). Build your E3 foundation first, then add E5 for automated risk response.

How many Conditional Access policies can a tenant have?#

The tenant-wide limit is 240 policies across all states — enabled, disabled, and report-only. Report-only policies count against this limit. Stale test policies and abandoned report-only configurations accumulate quickly in active tenants. Set a quarterly cleanup task. If you hit the limit, consolidate per-app policies into grouped resource policies and use filter-for-applications instead of manual app lists. Source: Microsoft Learn, Plan a Conditional Access deployment, 2025.

What replaced “Require approved client app” in Conditional Access?#

“Require application protection policy” (Intune MAM) is the replacement, effective March 2026. It provides the same data loss protection — and more. Unlike “Require approved client app,” it works on both enrolled and unenrolled devices. The only exception: Kaizala, Skype for Business, and Visio don’t support app protection policy and require the legacy control where still applicable. Migrate any policies using only “Require approved client app” before the control stops functioning.

What is the 2026 “All resources” enforcement change?#

Between March and June 2026, Microsoft is rolling out a change that enforces CA policies targeting “All resources” against OIDC-scope sign-ins, even when the policy contains resource exclusions. If your tenant has a catch-all MFA policy with specific app exclusions, users accessing excluded apps via OIDC flows may start receiving unexpected MFA challenges. Audit all “All resources” policies with exclusions in the Entra admin centre now. Source: Microsoft Learn, Common security policies for Microsoft 365, retrieved 2026-05-22.

What to Do This Week#

Conditional Access isn’t complex. It’s just consequential — a mistake in a broad policy affects everyone in your tenant at once. That’s why the phased approach and report-only validation aren’t optional steps you can skip when you’re in a hurry.

Start with Phase 1. Block legacy authentication. Secure the MFA registration page. Require phishing-resistant MFA for your privileged admins. Those three policies, run in report-only for a week and then enforced, close more attack surface than most security projects that take six months.

For the automation path, use New-MgIdentityConditionalAccessPolicy and build your policies as code from day one. When your CA baseline is in a script, rolling it out to a new tenant takes minutes, reviewing changes takes a code review, and recovering from a misconfiguration takes a single parameter change.
