Purview Endpoint DLP: Block Data Exfiltration from Windows Devices

Configure Purview Endpoint DLP on Windows 10/11 to block data exfiltration. Covers device onboarding, SIT conditions, block actions, and Activity Explorer.
May 23, 2026 · 12 min · Alexander Schirmer

The IBM Cost of a Data Breach Report 2024 puts the average breach cost at $4.88 million — and breaches traced to malicious insiders average nearly double that. DLP policies applied to email and SharePoint miss the most common exfiltration path: a user copying a file to a USB drive, printing it to a personal printer, or uploading it via an unmanaged browser. Microsoft Purview Endpoint DLP closes that gap. This post walks you through onboarding Windows devices, configuring policies with sensitive information type conditions, setting enforcement actions, and monitoring activity in Activity Explorer.

If you’re new to DLP policy anatomy — rules, conditions, and actions — read the Microsoft 365 DLP policy configuration guide first. That post covers Exchange, SharePoint, and Teams. This post covers the endpoint workload only.

Key Takeaways

  • Endpoint DLP requires Microsoft 365 E5 or the Microsoft 365 E5 Compliance add-on — E3 does not include it.
  • Policies apply to 13 monitored activities including USB copy, print, cloud upload, Bluetooth, clipboard, and Windows Recall snapshots.
  • Three enforcement modes — Audit, Block with override, and Block — let you roll out safely without locking down users immediately.
  • Activity Explorer gives you a searchable log of every policy match at the device level, filterable by activity type, user, and location.

How Endpoint DLP Differs from Exchange and SharePoint DLP#

Standard DLP policies in Microsoft Purview apply at the service level: Exchange Online inspects email in transit, SharePoint and OneDrive inspect files at rest and in motion, and Teams inspects messages and shared files. All of that inspection happens server-side. The endpoint workload is different.

Endpoint DLP installs a policy enforcement agent on the Windows device itself. That agent monitors file operations locally and enforces policy before data leaves the machine — not after it reaches a cloud service. This means it can intercept a USB copy operation before the file is ever written to the drive, and it can block a print job before it leaves the print spooler. It operates even when the device is offline. Source: Microsoft Learn — Learn about Endpoint data loss prevention, retrieved 2026-05-23.

Two further differences worth noting. First, file detection uses MIME type inspection, not just file extension. Renaming financial-report.xlsx to picture.jpg does not evade detection — Purview reads the file header and applies SIT matching to the actual content. Second, Endpoint DLP applies at the application layer for print and clipboard operations, which means policy applies whether the user prints from Word, Chrome, or a custom application.

Licensing: What E3 Gives You and What It Does Not#

Microsoft 365 E3 includes the standard DLP workloads: Exchange Online, SharePoint Online, OneDrive, and Teams. It does not include Endpoint DLP. This is a firm licensing boundary, not a configuration limitation.

Microsoft 365 E5 includes Endpoint DLP as part of the Microsoft 365 E5 Compliance suite. If your organisation is on E3, the Microsoft 365 E5 Compliance add-on makes Endpoint DLP available without upgrading the full E5 license. The add-on is licensed per user and must be assigned to every user whose devices will be subject to Endpoint DLP policies.

Microsoft 365 Business Premium also includes Endpoint DLP for organisations with fewer than 300 users, making it accessible to mid-market customers without a full E5 investment.

Onboarding Windows Devices to Microsoft Purview#

Before any Endpoint DLP policy can apply, devices must be onboarded to Microsoft Purview. Onboarding registers the device with the Purview compliance service and deploys the monitoring agent. Supported platforms include Windows 10 (version 20H2 or later), Windows 11, Windows Server 2019, and Windows Server 2022. macOS Catalina 10.15.7 and later is also supported.

Microsoft provides five onboarding methods:

1. Local script — Downloads a script from the compliance portal and runs it manually on each device. Suitable for testing with 10 or fewer devices. Not scalable to production.

2. Group Policy — Deploys the onboarding package via GPO. Suitable for on-premises Active Directory environments without Intune. Requires a package downloaded from the compliance portal.

3. Microsoft Intune / MDM — The preferred method for cloud-managed fleets. Create a device configuration profile of type “Endpoint Detection and Response” using the onboarding package from Purview. This is the same package used for Microsoft Defender for Endpoint, and if MDE is already deployed the devices may already be onboarded.

4. Microsoft Defender for Endpoint shared onboarding — If your tenant has MDE deployed, enabling the Purview integration in Microsoft Defender settings automatically onboards MDE-enrolled devices to Purview without additional configuration. This is the fastest path for organisations already running Defender.

5. Microsoft Endpoint Configuration Manager (SCCM/ConfigMgr) — Suitable for large hybrid environments. Deploys the onboarding package as a configuration item.

After onboarding, devices appear in the Microsoft Purview compliance portal → Settings → Device onboarding → Devices list. Allow up to 24 hours for a newly onboarded device to become eligible for policy assignment.

Virtualised environments require additional planning. In Azure Virtual Desktop, Windows 365 Cloud PC, and Citrix environments, USB devices are often presented as network redirected drives rather than physical USB devices. Endpoint DLP can still intercept these operations via the “Copy to network share” or “Copy to removable media” activities, but you must verify the activity type matches your policy conditions. Test policy behaviour explicitly in your VDI environment before enforcing block actions.

Configuring Your First Endpoint DLP Policy#

Endpoint DLP policies are created and managed in the Microsoft Purview compliance portal → Data loss prevention → Policies → Create policy. The workflow is the same as for Exchange or SharePoint policies: choose a template or create a custom policy, define the rules, and set the scope.

Scope the policy to devices. On the “Locations” page, enable Devices and disable any workloads you do not want this specific policy to cover. A dedicated endpoint policy is easier to tune than one combined policy that spans Exchange, SharePoint, and Devices.

Set your conditions using sensitive information types (SITs). The condition “Content contains” lets you select from over 300 built-in SITs or custom SITs your organisation has defined. Effective starting conditions for most organisations include:

  • Credit card numbers
  • EU debit card numbers
  • UK National Insurance numbers (or equivalents for your region)
  • Azure storage connection strings
  • Generic password in file

Combine multiple SITs using Any of these or require a minimum instance count to reduce false positives. A rule that triggers on a single credit card number in a file will generate far more alerts than one requiring five or more distinct credit card numbers.

Apply SIT confidence thresholds. Each SIT returns a high, medium, or low confidence match. For block actions on endpoint devices, using only high confidence matches reduces the risk of blocking legitimate business operations. Start with high confidence during the audit phase before expanding to medium confidence once your baseline is established.

Endpoint DLP Actions#

Each rule in an Endpoint DLP policy allows a different action per endpoint activity. Unlike Exchange DLP — where you typically either allow or block — Endpoint DLP supports three enforcement modes:

Audit only — The activity is permitted but logged to Activity Explorer. No user notification. Use this mode when onboarding new device groups or adding new SIT conditions. Run in audit mode for at least two weeks before moving to restrictive actions.

Block with override — The activity is blocked and the user sees a policy tip. The user can provide a business justification and proceed. The override is logged. This mode balances enforcement with business continuity: users who have a legitimate reason can still complete the operation, and you see the override rate in reporting.

Block — The activity is blocked without an override option. The user sees a policy tip explaining why the action was blocked. Use this mode for your highest-risk activities once audit data confirms low false-positive rates.

Per-activity granularity means you can apply different modes to different activities within the same rule. A common pattern: set USB copy to Block, print to Block with override, and cloud upload to Audit while you build your approved service domains list.

Endpoint-Specific Settings#

The Endpoint DLP settings page (compliance portal → Data loss prevention → Endpoint DLP settings) controls global device behaviour that applies across all endpoint policies.

USB device groups — Define groups of approved USB devices by vendor ID, product ID, serial number range, or a combination. Use these groups in policy conditions to allow specific approved devices (for example, corporate-issued encrypted drives) while blocking all others. Source: Microsoft Learn — Configure endpoint DLP settings, retrieved 2026-05-23.

Printer groups — Define groups of approved printers by printer name pattern, port, or network share path. Use these in policy conditions to allow printing to corporate print servers while blocking local or Bluetooth printers.

Restricted apps and app groups — List applications that should trigger a block or audit event when they access a sensitive file, regardless of what the application does with the data. This is useful for blocking known data exfiltration tools, personal cloud sync clients (Dropbox, personal OneDrive), or screenshot utilities.

Unallowed browsers — Specify browsers that should be blocked from uploading sensitive files. By default, users can use any browser to upload to cloud services. Configuring unallowed browsers forces users to work within browsers that are covered by the Purview browser extension.

File path exclusions — Exclude specific local paths from policy evaluation. Use this for development directories or backup agents that need unrestricted file access. Be conservative with exclusions — they are a common misconfiguration that creates blind spots.

Sensitive service domains — For the “Upload to cloud services” activity, define which domains are monitored. A corporate SharePoint tenant should typically be excluded; consumer cloud storage services should be monitored.

Policy Tips on Windows Devices#

Policy tips in Endpoint DLP appear as a dialog or notification in the application where the user is performing the restricted activity — in Explorer for USB/network share operations, in the print dialog for print activities, and in the browser for cloud uploads.

Policy tips can be customised with:

  • A custom notification message explaining the policy
  • A link to a company policy page or IT support portal
  • Business justification options (free text or a predefined list for block-with-override rules)

Effective policy tip messages name the specific SIT detected (“Your document contains credit card numbers”), reference the business policy (“Company policy restricts copying financial data to external devices”), and provide a clear path for exceptions (“Contact IT if you need a temporary exception”). Vague messages (“This operation is not permitted”) lead to help desk tickets rather than informed user decisions.

Monitoring with Activity Explorer and PowerShell#

Activity Explorer (compliance portal → Data loss prevention → Activity Explorer) is the primary monitoring surface for Endpoint DLP events. Each logged event includes: timestamp, user, device name, activity type, file name and path, SIT detected, confidence level, policy name, rule name, and the action taken (audit, blocked, overridden).

Filter by Activity type = Endpoint activities and Policy = [your policy name] to isolate endpoint DLP events from Exchange and SharePoint DLP noise. Export to CSV for trend analysis or to validate that your block actions are triggering on the expected file types.

To verify your Endpoint DLP policies programmatically, use the Security & Compliance PowerShell module:

#Requires -Modules ExchangeOnlineManagement

Connect-IPPSSession -UserPrincipalName admin@contoso.onmicrosoft.com

# List all DLP policies scoped to the Endpoint workload
$params = @{
    ErrorAction = 'Stop'
}
Get-DlpCompliancePolicy @params |
    Where-Object { $_.Workload -like '*Endpoint*' } |
    Select-Object Name, Mode, Priority, Workload |
    Format-Table -AutoSize

# Retrieve rules for a specific policy
$policyParams = @{
    Policy      = 'Endpoint - Block USB Exfiltration'
    ErrorAction = 'Stop'
}
Get-DlpComplianceRule @policyParams |
    Select-Object Name, ContentContainsSensitiveInformation, BlockAccess

Connect-IPPSSession authenticates to the Security & Compliance endpoint rather than Exchange Online. If you are already connected via Connect-ExchangeOnline, you still need a separate Connect-IPPSSession call for compliance cmdlets. The ExchangeOnlineManagement module (v3.x or later) provides both connection commands.

Use Get-DlpCompliancePolicy during rollout to confirm that policies are in the expected mode (-Mode TestWithNotifications for audit, Enforce for enforcement) before enabling block actions across your device fleet.

Conclusion#

Microsoft Purview Endpoint DLP fills the gap that service-level DLP policies leave open: file operations that happen entirely on the device, before any cloud service is involved. The rollout sequence that minimises disruption is: onboard devices → deploy in audit mode → review Activity Explorer for two weeks → tune SIT conditions and exclusions → enable block with override → promote high-confidence, high-risk activities to full block.

The licensing requirement is a hard constraint: E3 tenants need the E5 Compliance add-on before a single Endpoint DLP policy can be enforced. Verify licensing in the compliance portal before beginning device onboarding.
